# Zejtron v5.0.0 Design Specification

This document defines the design for Zejtron v5.0.0. It is a specification
only. No runtime behavior changes are implied until implementation begins on
the designated branch.

## 1. Purpose and Scope

Zejtron v5.0.0 is a major release that extends the introspection surface of
the tool into three new diagnostic namespaces: shell environments, network
state, and git repository inspection. The release is justified as major because
it introduces new top-level command groups that expand Zejtron beyond its
original process, port, and file scope into adjacent operational domains.

v5.0.0 does not change any existing command behavior. All ten current
commands (path, recent, port, proc, holds, touch, why, env, service, doctor)
must produce identical output to v2.4.6. This is a hard requirement.

## 2. Design Principles

### 2.1 Read-Only Invariant

All new namespaces must be read-only. They must inspect, query, and report
without modifying any state on the host system. This invariant is
non-negotiable and has governed every Zejtron command since v1.0.0.

### 2.2 Minimal Dependency Footprint

v5.0.0 must not introduce high-risk or heavy dependencies. New dependencies
must be justified with a written rationale in the commit message and reviewed
against the existing dependency policy in RULES.md. The current runtime
dependency set (chrono, clap, ctrlc) represents a deliberate baseline. Any
addition must carry clear value that cannot be achieved through the standard
library or existing dependencies.

### 2.3 Single Binary

All new namespaces must compile into the same single binary. No plugins, no
dynamic loading, no external runtime components. The binary must continue
to build with `cargo build --release --locked` and produce a statically
optimized artifact.

### 2.4 LOC Discipline

All existing LOC rules apply. No source file may exceed 1000 lines of code.
Each new namespace module should follow the same decomposition pattern
established by the port module: a directory containing focused submodules for
model, parsing, and rendering responsibilities.

### 2.5 Low Maintenance Burden

New namespaces must not increase CI complexity, build time, or dependency
update surface beyond acceptable margins. The weekly dependency refresh
workflow must continue to operate correctly. The release and AUR sync
workflows must not require structural changes.

## 3. Proposed Namespaces

### 3.1 `zejtron shell`

Inspect the current shell process and its environment context.

**Scope:**

- Identify the current shell type (bash, zsh, fish, etc.) by inspecting
  parent process lineage and environment variables.
- Report shell configuration file locations (e.g. .bashrc, .zshrc) and
  whether they exist and are readable.
- Show shell options and flags currently in effect.
- Report shell function definitions matching a given pattern.
- Display shell alias definitions matching a given pattern.

**Out of scope:**

- Executing shell commands or evaluating expressions.
- Modifying shell configuration.
- Sourcing or reloading configuration files.

**Design notes:**

Shell inspection must work by reading /proc entries for the current process
and its ancestors, combined with parsing known configuration file paths. It
must not require spawning a subshell or evaluating shell-specific syntax. This
approach keeps the implementation portable across shell types and avoids
injection risks.

### 3.2 `zejtron net`

Inspect network interface state and routing information.

**Scope:**

- List network interfaces with status (up/down), MTU, MAC address, and
  assigned addresses (IPv4 and IPv6).
- Show the routing table in a readable format.
- Report DNS resolver configuration from /etc/resolv.conf.
- Inspect firewall rule counts (where supported) without modifying rules.
- Show network namespace information (where available).

**Out of scope:**

- Modifying interface configuration (ip link set, ifconfig).
- Adding or removing routes.
- Modifying firewall rules.
- Packet capture or traffic analysis.
- Bandwidth measurement or rate limiting.
- DNS queries or resolution (as opposed to reading resolver configuration).

**Design notes:**

Network inspection must read from /proc/net and sysfs (/sys/class/net).
It must not depend on `ip`, `ifconfig`, `ss`, or other external commands.
Parsing must handle both IPv4 and IPv6 address formats. The implementation
should follow the same model/parser/render decomposition as the port module.

### 3.3 `zejtron git`

Inspect git repository state without performing repository operations.

**Scope:**

- Report current branch, HEAD commit, and remote tracking status.
- Show working tree status (clean, modified, untracked) in a concise format.
- List branches (local and remote) with optional filtering.
- Show recent commit history in a configurable short format.
- Report stash list (if any).
- Show submodule status.

**Out of scope:**

- Cloning, fetching, pulling, or pushing.
- Committing, staging, or amending.
- Branch creation, deletion, or merging.
- Conflict resolution.
- Any write operation on the repository.

**Design notes:**

Git inspection must invoke `git` as an external process and parse its output.
Zejtron must not link against libgit2 or attempt to interpret the .git
directory directly. This keeps the dependency footprint minimal and delegates
correctness of git internals to the git binary itself, which is expected
to be present on any system where git introspection is useful. Error handling
must account for environments where git is not installed and must degrade
gracefully with a clear message.

## 4. Command Taxonomy

v5.0.0 introduces a namespace layer between the binary name and the command
verb. The full command taxonomy becomes:

```
zejtron <namespace> <verb> [options]
```

Existing commands remain flat because they represent the core introspection
surface that has been stable since v1.0.0. Refactoring them into namespaces
(e.g. `zejtron proc inspect`) is explicitly out of scope for v5.0.0 to
avoid breaking changes. The proposed namespaces are additive only.

| Command | Type | Namespace |
|---------|------|-----------|
| path    | existing | (none) |
| recent  | existing | (none) |
| port    | existing | (none) |
| proc    | existing | (none) |
| holds   | existing | (none) |
| touch   | existing | (none) |
| why     | existing | (none) |
| env     | existing | (none) |
| service | existing | (none) |
| doctor  | existing | (none) |
| shell   | new v5   | shell | Implemented in v3.0.0 |
| net     | new v5   | net   | Implemented in v5.0.0 |
| git     | new v5   | git   | Implemented in v5.0.0 |

The Clap command tree in cli.rs must accommodate both flat and namespaced
commands. The implementation should use Clap's subcommand nesting to define
the new namespaces without restructuring existing flat commands.

## 5. Module Layout

The source tree structure for v5.0.0 namespaces follows the pattern
established by the port module:

```
src/
  shell/
    mod.rs        -- public entry point, run()
    model.rs      -- data types for shell info
    parser.rs     -- /proc and config file parsing
    render.rs     -- output formatting
  net/
    mod.rs        -- public entry point, run()
    model.rs      -- data types for interfaces, routes, DNS
    parser.rs     -- /proc/net and sysfs parsing
    render.rs     -- output formatting
  git/
    mod.rs        -- public entry point, run()
    model.rs      -- data types for repo state
    parser.rs     -- git output parsing
    render.rs     -- output formatting
```

Each module exposes a single `run()` function that accepts flags and produces
output. This is the same pattern used by all existing commands. Each
submodule carries the SPDX license header and includes unit tests.

## 6. Testing Rules

All new namespaces must satisfy the following testing requirements:

- Unit tests in each source file covering parsing, validation, and rendering
  logic. Tests must exercise edge cases: empty input, malformed data, missing
  files, and permission errors.
- Integration smoke tests in check.sh covering basic invocation of each new
  command with expected output patterns.
- Error-case tests in check.sh verifying that invalid flags, missing
  dependencies (e.g. git not installed), and permission failures produce clear
  error messages without panicking.
- No test may modify system state. Tests must clean up temporary files and
  restore environment variables after execution.

## 7. LOC Rules

The existing LOC rules from RULES.md apply in full:

- No source file exceeds 1000 LOC.
- main.rs stays in the 100-300 LOC range.
- Each submodule has a clear, single responsibility.
- When any file approaches the 1000 LOC threshold during development, it
  must be split before the PR is submitted.

Given the current largest file (proc.rs at 921 LOC), v5.0.0 development
should also evaluate whether proc.rs should be decomposed proactively as part
of the release preparation. This is a maintenance decision, not a feature
decision.

## 8. CI and Release Rules

v5.0.0 must not change the CI workflow structure. The existing ci.yml,
release.yml, aur.yml, maintenance.yml, and docs-ci.yml workflows must
continue to function without modification to their job definitions or
permission model.

Specific CI requirements for v5.0.0:

- All new source files must be covered by the existing ci.yml path triggers
  (they match **/*.rs already).
- check.sh must be extended to include smoke tests for new commands.
- codespell must pass on all new documentation files.
- The release workflow must produce multi-arch binaries that include the new
  namespaces without increase in binary size exceeding 15% of the v2.4.6
  baseline.

## 9. Dependencies

New dependencies for v5.0.0 namespaces must be evaluated against the
following criteria:

- Is the functionality achievable through the standard library alone? If
  yes, no new dependency is justified.
- Is the dependency mature, actively maintained, and audited?
- Does the dependency add more than 500KB to the release binary?
- Does the dependency pull in transitive dependencies that create supply
  chain risk?

A preliminary assessment:

- `shell` namespace: No new dependencies expected. Reads /proc and parses
  text files.
- `net` namespace: No new dependencies expected. Reads /proc/net and
  sysfs.
- `git` namespace: No new dependencies expected. Invokes git binary and
  parses stdout.

If the implementation reveals that a new dependency is unavoidable, it must
be documented with a written justification and approved before merging.

## 10. Release Gates

v5.0.0 must pass all release gates defined in [ROADMAP.md](../ROADMAP.md)
before tagging. In addition:

- All new commands must have complete help text that is consistent with
  their behavior.
- README.md must be updated with a command overview table that includes the
  new namespaces.
- CHANGELOG.md must document the v5.0.0 release with a clear summary of what
  changed.
- No known regressions against v2.4.6 in any existing command.
- Soak testing on at least two distinct Linux distributions (one Arch-based,
  one Debian-based) must be completed and documented.

## 11. Risks and Mitigations

### 11.1 Scope Creep

Risk: During implementation, pressure to add commands beyond the three
proposed namespaces.

Mitigation: The design freeze in Phase 1 is binding. Any command not in
docs/v5.md must go through a new design cycle for a future minor release.

### 11.2 Binary Size

Risk: Adding three namespaces increases binary size beyond acceptable
thresholds.

Mitigation: The 15% size increase ceiling is enforced at release time. If the
ceiling is exceeded, the implementation must be optimized or the scope reduced.

### 11.3 proc.rs LOC Pressure

Risk: proc.rs at 921 LOC may exceed 1000 LOC if unrelated changes are made
during v5 development.

Mitigation: Evaluate proc.rs for proactive decomposition during Phase 2.
Splitting proc.rs into submodules (following the port pattern) should be
considered early.

### 11.4 Git Binary Availability

Risk: `zejtron git` requires an external `git` binary that may not be
present.

Mitigation: The command must detect missing git and produce a clear error
message. This is not a failure of Zejtron; it is an expected environmental
constraint. The error message should suggest installing git and must not
panic or crash.

## 12. Timeline

No timeline is committed. The staged roadmap in ROADMAP.md defines the
sequence of phases, not deadlines. Each phase must complete fully before the
next begins. The project exits stabilization mode only when the design
freeze is complete and implementation is authorized.
